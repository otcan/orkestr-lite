# Loom demo script

Target length: **2 minutes 30 seconds to 2 minutes 45 seconds**.

This is a spoken product story, not a voice-over for a slideshow. Record the
real Orkestr interface in Loom, move through the screens below, and speak in
your normal voice. Pause the recording between screens if loading takes time.
Do not pretend that edited waiting time happened live.

## Before recording

- Record at 1440p or 1080p with browser zoom at 100%.
- Close personal tabs, notifications, bookmarks, and unrelated applications.
- Prepare the completed research run before pressing Record.
- Open these screens in order: Setup, Chat, Desk, Files, WhatsApp, Timers, Chat.
- Keep the cursor still while speaking. Move it only to point at something.
- Use your camera for the first and last sentence if that feels natural. The
  product should fill the screen for the rest.

## Recording script

### 0:00 to 0:22 | Start with the reason

**Show:** Orkestr Chat, or your camera for the first sentence and then Chat.

**Say:**

> Codex is already very good at doing the work. The problem I kept hitting was
> everything around it: keeping it available, continuing from my phone,
> scheduling work, and having a browser I could inspect or take over. I was
> stitching together terminals, Docker, WhatsApp, and timers. So I built Orkestr
> Lite.

### 0:22 to 0:40 | Explain the product in one sentence

**Show:** Setup. Point once at Codex connected and once at WhatsApp connected.

**Say:**

> Orkestr is a self-hosted operating layer around Codex. One conversation gets a
> persistent Ubuntu workstation. Work can enter from the browser, WhatsApp, or a
> schedule. Here, my Codex account and optional WhatsApp account are connected.

### 0:40 to 1:02 | Start real work

**Show:** Chat. Briefly point at GPT-5.6 and the research request. Scroll only
enough to show the request and completed response.

**Say:**

> This is a real GPT-5.6 run. I asked Codex to compare three agent runtimes from
> their official documentation and produce a cited Markdown report and a readable
> HTML version. The topic is only the example. The point is that one request can
> use the same browser, files, terminal, and context.

### 1:02 to 1:25 | Show the workstation

**Show:** Desk. Let the generated HTML report remain visible. Point at Pause
Codex and Full Screen, but do not click them.

**Say:**

> This is the live Ubuntu workstation where Codex did the work. Chromium is
> showing the report it created. I can watch, pause Codex, take control, or close
> my browser and come back later without losing the workstation or its browser
> state.

### 1:25 to 1:42 | Show useful output

**Show:** Files. Select the Markdown file, then point at Download and Send to
WhatsApp.

**Say:**

> The output is not trapped in chat. The Markdown and HTML files stay in the
> workspace. I can inspect them, download them, keep working, or send an allowed
> file directly to WhatsApp.

### 1:42 to 2:04 | Continue from WhatsApp

**Show:** The sanitized WhatsApp conversation. Keep the request, Working reply,
completed reply, and returned document visible.

**Say:**

> Then I continued from my phone. I asked for a sourced follow-up through my own
> WhatsApp self-chat. Orkestr sent it into the same Codex conversation, kept the
> research context, and returned the updated Markdown file. This is one operator,
> not a separate bot conversation.

### 2:04 to 2:27 | Schedule the next action

**Show:** Timers. Point at the schedule selector and the existing weekly watch,
then click or point at Run now. Switch to Chat and show the scheduled result.

**Say:**

> Work can also start once, every few hours, daily, weekly, or from a cron
> schedule. Here the weekly source review enters the same queue. Run now uses the
> real schedule path, and the result returns to the same history and WhatsApp.
> Manual and recurring work do not split into disconnected systems.

### 2:27 to 2:48 | End with the thesis

**Show:** Return to Chat, then open the repository or use your camera for the
last two sentences.

**Say:**

> The same loop can support outreach, job applications, messaging, or recurring
> checks. Codex supplies the intelligence. Orkestr supplies the persistent
> machine, the inputs, the schedule, and the path back to the human. It is open
> source, self-hosted with Docker, and built with Codex and GPT-5.6. Orkestr turns
> a Codex session into a workstation that stays on.

## If you need to shorten it

Cut the second sentence in the Files section and the first sentence in the
closing section. Do not speed up the recording. A calm 2:35 video is stronger
than a rushed 2:10 video.

## Do not say

- that Orkestr replaces Codex;
- that Codex cannot already work remotely;
- that LinkedIn, job boards, or other external services are built-in
  integrations;
- that the whole demo happened in real time if you paused or cut waiting time;
- that the Desk is a virtual machine if the submitted runtime is containerized.
